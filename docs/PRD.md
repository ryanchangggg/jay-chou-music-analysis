## 1. 项目概述

### 1.1 项目背景

周杰伦是华语流行音乐最具标志性的音乐人，从 2000 年首张专辑《Jay》到 2026 年的《太阳之子》，横跨 26 年、16 张录音室专辑、174 首作品。项目已收集了完整的演唱会音乐特征数据（Spotify Audio Features）、歌词文本、专辑封面、人工标注风格标签等信息，但尚未进行系统性的分析和可视化呈现。

### 1.2 项目目标

构建一个**互动式数据叙事网站**，以周杰伦 26 年职业生涯为线索，通过数据可视化回答以下核心问题：

- 周杰伦的音乐风格随时代发生了怎样的演变？
- 哪首歌/哪张专辑数据上最"特别"？为什么？
- 方文山 vs 周杰伦作词，在音乐特征上有无可量化的差异？
- 中国风歌曲是真的有可识别的"音频指纹"吗？
- 26 年来听众口味的变化能否从数据中读出？

**交付物**：
1. 一个可直接访问的静态/轻后端 Web 应用
2. 包含 5-7 个交互式数据页面的数据叙事
3. Jupyter Notebook 作为分析层（src/notebooks）
4. README + 项目文档，适合放入技术作品集

### 1.3 核心指标

| 指标 | 目标 |
|---|---|
| 加载速度 | 首屏 < 2s |
| 交互响应 | 图表切换/筛选 < 500ms |
| 页面覆盖 | 7 个分析页面全部可交互 |
| 移动端适配 | 基础阅读体验可用 |

## 2. 用户画像与使用场景

### 2.1 目标用户

**P1 — 数据科学 / 数据可视化求职者（主要用户）**

- 活跃于 GitHub，用个人项目展示数据叙事和全栈能力

- 关注点：代码质量、分析深度、叙事流畅度、技术栈选择

- 通过浏览发现冷知识（"原来《晴天》不是最悲伤的歌"）

**P3 — 音乐产业 / 音乐研究者**

- 对音乐特征与流行度/时代趋势感兴趣

## 3. 数据故事线与章节安排

### 3.1 整体叙事框架

**叙事主题**：一个人、两种身份、三种风格的 26 年

- **钩子**：26 年，16 张专辑，174 首歌。周杰伦从青涩少年到华语乐坛天花板，数据告诉我们什么？
- **上升行动**：从整体画像 → 时间演变 → 风格解构 → 创作团队分析 → 歌词深度
- **高潮**：聚类/异常检测揭示"最不周杰伦"的歌
- **回落**：Top N 榜单 + 互动探索
- **结尾**：总结 + 数据卡片唤起行动

### 3.2 七章安排

| 章 | 标题 | 核心问题 | 篇幅 | 交互等级 |
|---|---|---|---|---|
| 1 | 全景 | 26 年职业生涯的整体面貌？专辑销量 vs 口碑曲线？ | 一屏总览 | ★★ |
| 2 | 时代之声 | 音乐特征（能量、舞动性、声学性）随年代如何变化？ | 滚动长页 | ★★★ |
| 3 | 风格指纹 | 不同风格标签（中国风、情歌、HipHop）的音频指纹是什么？ | 固定布局 | ★★★★ |
| 4 | 创作密码 | 周杰伦 vs 方文山 vs 其他作词人，数据差异？ | 对比布局 | ★★ |
| 5 | 歌词之眼 | 歌词主题聚类、高频词变迁、情感分析 | 滚动长页 | ★★★ |
| 6 | 异类 | 聚类 + 异常检测：最"不周杰伦"的歌曲是哪首？ | 互动仪表盘 | ★★★★★ |
| 7 | 探索器 | 自由筛选 + 对比任意歌曲的全维度雷达图 | 全交互 | ★★★★★ |

## 5. 页面级功能规格

### 5.1 首页 — 全景

**核心问题**：周杰伦 26 年职业生涯的整体面貌如何？

**功能列表**：

| 功能模块 | 图表类型 | 数据字段 | 交互 |
|---|---|---|---|
| 顶部英雄区 | 大数字+动效 | 专辑数(16)、歌曲数(174)、跨度(26年)、平均流行度(63.5) | 数字滚动动效 |
| 专辑时间线 | 水平点线图/自定义 | 专辑名、年份、歌曲数、封面缩略图 | 悬停显示专辑封面+信息卡片 |
| 年度特征趋势 | 多线图 | 年份 vs 平均 energy/danceability/valence | 切换特征线，悬停读值 |
| 流行度柱状图 | 水平柱状图 | each song popularity | 排序切换（年份/流行度） |

**数据流**：从 master_dataset 按 year 聚合 → 计算每年均值 → 传入前端

**要回答的问题**：
- 周杰伦哪个时期最"高能"？
- 早期 vs 后期音乐风格有哪些质的差异？
- 最受欢迎/最冷门的歌曲是什么？

### 5.2 时代之声

**核心问题**：周杰伦的音乐特征随时间如何系统性漂移？

**功能列表**：

| 功能模块 | 图表类型 | 数据字段 | 交互 |
|---|---|---|---|
| 特征趋势总览 | 平行坐标/小多线图 | 所有 Spotify 特征按年份 | 筛选/高亮特定特征 |
| 专辑窗口对比 | 箱线图 | 每张专辑的 energy, danceability, acousticness 分布 | 悬停显示专辑详情 |
| 年代划分着色 | 散点图 (PCA/TSNE) | 按 decade 着色，x=danceability, y=energy | 悬停显示歌曲名 |
| 动画时间轴 | 点状分布动画 | 逐年所有歌曲散点，颜色=专辑 | 播放/暂停/拖动 |

**要回答的问题**：
- 2000-2004 的"黄金期"与后期有何不同？
- 《惊叹号》(2011)专辑特征是否真的是转折点？
- acousticness 的变化趋势反映了怎样的制作理念变化？

### 5.3 风格指纹

**核心问题**：不同风格标签（中国风、情歌、HipHop）在音频特征上是否有可量化的差异？

**功能列表**：

| 功能模块 | 图表类型 | 数据字段 | 交互 |
|---|---|---|---|
| 标签分布 | 雷达图（每个标签一张） | 各标签的 8 维特征均值 | 切换标签，查看叠加对比 |
| 风格聚类 | 2D 散点 (PCA/TSNE) | 所有歌曲，点=风格标签 | 悬停显示歌曲信息，颜色=标签 |
| 特征差异检验 | 柱状图+误差线 | 每一特征上的组间均值±SE | 点击特征突出显示 |
| 歌曲分类器效果 | 混淆矩阵热力图 | 预测标签 vs 真实标签 | 悬停显示精确数值 |

**要回答的问题**：
- 中国风歌曲真的 acousticness 更高？tempo 更慢？
- 情歌 vs R&B 在特征空间上是否可区分？
- 哪个标签在音频特征上最"独特"？

### 5.4 创作密码

**核心问题**：不同作词人/作曲人合作下，歌曲的音频特征有何差异？

**功能列表**：

| 功能模块 | 图表类型 | 数据字段 | 交互 |
|---|---|---|---|
| 合作网络 | 力导向图 | 节点=作词人/专辑，边=合作 | 拖拽节点，悬停显示合作歌曲数 |
| 特征对比 | 分组柱状图 | 方文山 vs 周杰伦 vs 其他，按特征分组 | 切换特征维度 |
| 作词人风格画像 | 迷你雷达图矩阵 | 每位作词人作品平均特征 | 点击放大 |
| 时间分布 | 堆叠面积图 | 各作词人合作频率随时间变化 | 悬停读值 |

**数据字段关联**：master_dataset.lyricist + audio features

**要回答的问题**：
- 方文山的词配上周杰伦的曲，在音频特征上有无统计显著差异？
- 黄俊郎 vs 方文山，谁的词更"暗黑"？
- 周杰伦自己作词的作品和他作曲的有何不同？

### 5.5 歌词之眼

**核心问题**：歌词文本中隐藏了怎样的情感结构和话题演变？

**功能列表**：

| 功能模块 | 图表类型 | 数据字段 | 交互 |
|---|---|---|---|
| 高频词云 | 词云 | 所有歌词 TF-IDF 前 80 词 | 可筛选年代/专辑，切换中英文 |
| 话题演化 | 话题分布堆叠图 | LDA/BERTopic 提取 5 个主题，按年分布 | 悬停显示主题关键词 |
| 情感时间线 | 折线图 | 每首歌情感极性（积极/消极/中性） | 筛选标签，悬停读值 |
| 歌词长度趋势 | 箱线图 | 每首歌歌词字数/行数逐年变化 | 悬停显示具体歌曲 |
| 高频词变迁 | 棒棒糖图 | "爱"、"泪"、"时间"等关键词频次趋势 | 选择特定关键词 |

**要回答的问题**：
- 方文山的歌词核心词汇 vs 周杰伦的？
- 有没有哪首歌的歌词情感与音频 valence 严重不一致？
- 歌词长度在变短还是变长？
- 前期"爱较多"还是后期？

### 5.6 异类

**核心问题**：哪些歌曲在统计学上是"最不周杰伦"的？

**功能列表**：

| 功能模块 | 图表类型 | 数据字段 | 交互 |
|---|---|---|---|
| 聚类 3D 可视化 | 3D 散点图 (Three.js/Plotly) | PCA 前 3 主成分，颜色=聚类标签 | 旋转/缩放/点击选中 |
| 异常得分排名 | 水平条形图 | 每首歌的 Isolation Forest / LOF 异常分 | 点击跳转至歌曲详情 |

| 聚类明细表 | 可排序表格 | 聚类编号、成员数、特征均值、代表曲 | 点击筛选 |

**要回答的问题**：
- 《双截棍》真的是异常值吗？
- 哪些歌"跳出"了它们所在专辑的风格？
- 不同专辑内部歌曲的异质性程度？

| loudness | float dB | -9.0 ~ -3.5 | 响度 | 特征分析 |

| lyrics | text | - | 歌词全文 | 歌词分析 |
| album_cover_url | string | - | 专辑封面 URL | UI 展示 |

### 6.2 分析模块清单

| 模块 | 算法/方法 | 输入字段 | 输出 |
|---|---|---|---|
| 描述性统计 | 均值、中位数、标准差、偏度 | 所有 audio features | 每张专辑/每年特征汇总表 |
| 时间序列分析 | 滚动窗口均值 + LOESS 平滑 | features + release_date | 趋势线 + 置信带 |
| 聚类分析 | KMeans (n=4~6, 手肘法确定 k) | scaled audio features | 聚类标签 + 簇心坐标 |
| 异常检测 | Isolation Forest (contamination=0.05) | scaled audio features | 异常得分 + 排名 |
| 降维可视化 | PCA → 2D/3D | scaled audio features | 主成分坐标 |
| 组间差异检验 | ANOVA / Kruskal-Wallis + post-hoc | features + tags/lyricist | p-value + effect size |
| 歌词分词 | Jieba 分词 + TF-IDF | lyrics | 高频词表 + 词云输入 |
| 主题建模 | LDA (n=5) / BERTopic | 分词后文本 | 主题-词分布 + 文档-主题分布 |
| 情感分析 | 知网/大连理工情感词典 + 极性打分 | 分词后文本 | 每首歌情感极性值 (-1 ~ 1) |
| 分类验证 | Random Forest (5-fold CV) | audio features → tag | 混淆矩阵 + 特征重要性 |

## 7. 技术栈与目录结构

### 7.1 推荐技术栈

#### 前端

| 层级 | 选型 | 理由 |
|---|---|---|
| 框架 | React 18 + Vite | 轻量、快速 HMR、TypeScript 友好 |
| 组件库 | None (Custom CSS) | 避免大依赖；Tailwind CSS 3 可选用于快速布局 |
| 数据可视化 | ECharts 5 + D3.js (少量) | ECharts 成熟的交互图表，D3 用于自定义可视化（雷达、力导向图） |
| 3D | Three.js (via React Three Fiber) | 3D 聚类散点图（异类页面） |
| 路由 | React Router v6 | SPA 导航 |
| 状态管理 | React Context + useReducer | 足够，无需 Redux |
| 样式 | Tailwind CSS 3 + CSS Modules | 快速原型 + 局部样式隔离 |
| 数据获取 | fetch / d3.csv | 静态 CSV 前端直接读取（预处理后 JSON） |

#### 后端（轻量）

| 层级 | 选型 | 理由 |
|---|---|---|
| 服务器 | Node.js + Express 5 (dev-server) | 同 previous projects，一致性 |
| API | REST (无 DB，读 JSON 文件) | 数据量小，无需 database |
| 数据处理 | Python 3 (已安装 Jieba/sklearn) | Numpy/Pandas/Sklearn |
| Notebook | Jupyter | 数据分析、可视化探索、算法验证 |

## 8. 数据流与开发路线图

### 8.1 数据流

```
[raw CSV] ──▶ Python 预处理 ──▶ [processed JSON] ──▶ Vite/React ──▶ 浏览器渲染
                                                          │
                                                          ├── 首页 / 静态数据
                                                          ├── 交互筛选 / 前端过滤
                                                          └── 探索器 / 动态视图
```

**详细流程**：

1. **数据摄入**：6 个原始 CSV 文件存入 `data/raw/`
2. **清洗合并**：`src/python/preprocess.py` 中，以 `song_name` 为主键，合并 master_dataset + lyrics, 统一字段名、处理缺失值、类型转换
3. **特征工程**：`src/python/features.py` — 标准化 8 维音频特征、计算衍生字段（valence-energy 象限、情绪维度等）
4. **分析管线**：
   - `clustering.py` → KMeans + PCA + Isolation Forest
   - `lyrics.py` → Jieba 分词 + TF-IDF + LDA
   - `sentiment.py` → 情感词典打分
   - `stats.py` → ANOVA 检验
5. **前端数据生成**：`notebooks/06_prepare_frontend_data.ipynb` 将所有分析结果输出为多个 JSON 文件，放入 `data/processed/`
6. **前端加载**：`useData` hook 在应用启动时 fetch 所有 JSON，存入 React Context；页面按需读取子集
7. **交互过滤**：所有筛选/排序在浏览器端执行（数据量 174 条，无需服务端）

## 9. 风险点与应对方案

### 9.1 数据版权与使用

| 风险 | 影响 | 应对方案 |
|---|---|---|
| 歌词版权问题 | 公开部署可能侵权 | 前端歌词展示使用片段 (< 30%) 或仅展示分析结果（词频、情感值）而非全文；歌词全文仅在本地 notebook 中可见 |
| 专辑封面版权 | 合规问题 | 使用 Wikimedia Commons 等公开来源的 URL，不本地存储；添加 Fair Use 声明 |
| Spotify 数据使用 | 超出 TOS | 仅使用公开 API 可获取的 Audio Features（无公开视频/音频），在 README 中注明数据来源 |

### 9.2 Spotify API 限制

| 风险 | 影响 | 应对方案 |
|---|---|---|
| 歌曲 ID 不持久 | 无法增量更新 | 当前数据已全部下载完毕，项目作为静态分析无更新需求 |
| play_count_est 准确性 | 分析偏差 | 在页面中标注"估算值"，不以此作为核心分析结论 |
| Audio Features 实时获取 | 限速 | 现有数据为本地 CSV，无 API 调用需求 |

### 9.3 歌词数据处理

| 风险 | 影响 | 应对方案 |
|---|---|---|
| 中文分词准确度 | 词云/主题模型质量 | 使用 Jieba 精确模式 + 自定义词典（加入周杰伦歌曲特有词汇） |
| 简体/繁体混合 | 词频统计偏差 | 统一使用 opencc 转简体后再分析 |
| 情感词典覆盖 | 情感分析准确度 | 使用大连理工大学情感词汇本体库 + 知网 Hownet；标注置信度供读者参考 |
| 歌词片段的上下文 | 情感/主题误判 | 对短歌词（< 50 字）跳过情感分析 |

### 9.4 技术风险

| 风险 | 影响 | 应对方案 |
|---|---|---|
| ECharts + Three.js 在同页面渲染冲突 | 性能/渲染异常 | 不同页面隔离渲染，不共用 canvas 上下文 |
| 174 首歌 PCA 聚类结果不稳定 | 学术严谨性 | 在页面中标注"算法参数"说明，提供 silhouette score |
| 移动端图表触控体验 | 用户体验差 | 使用 ECharts 的触控配置项，关键页面（探索器）桌面优先 |
| 静态站点 SEO | 可发现性 | 使用 React Helmet 设置各页面 meta title/description |

### 9.5 数据质量问题

| 风险 | 影响 | 应对方案 |
|---|---|---|
| has_lyrics=no 的 13 首歌 | 歌词分析完整性 | 在歌词页面标注覆盖率（161/174），不在无歌词歌曲上做主题推断 |
| tags 为人工标注 | 主观偏差 | 在页面中注明"人工标注"，允许用户自行筛选/过滤 |
| 部分早期歌曲播放量估算偏低 | 趋势偏移 | 使用 popularity（标准化值）而非 play_count_est 做时间维度分析 |
